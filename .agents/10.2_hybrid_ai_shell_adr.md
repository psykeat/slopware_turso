# ADR 10.2: Shared AI Overlay Shell

## Status

DECIDED

## Decision

slopware uses one shared AI overlay host for all AI-assisted workflows.

## Why

- The shell must have zero permanent layout footprint.
- The overlay must follow the same focus and Escape lifecycle as the rest of the platform.
- AI should be context-bound and command-driven, not screen-specific.

## Contract

- The overlay is opened through registered commands in `CommandProvider`.
- The overlay reads the active focus context from the platform focus manager.
- The overlay renders review payloads produced by server tasks.
- The overlay does not own business mutation logic.

## Current implementation shape

- `AiOverlayHost` is the shared shell implementation.
- The overlay is triggered from `Alt+A` and contextual toolbar actions.
- Task scopes are registered in the client capability registry.
- The server produces interpretation, resolution, review, validation, and apply responses.

## Scope boundary

The shared shell is generic. Mail is the first scope, not the shell itself.

## Legacy note

Older `plan`/`plans/:id` compatibility routes remain in the codebase for transition only. They are not the preferred architecture for new work.
