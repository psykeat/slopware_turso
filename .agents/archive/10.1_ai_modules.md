# AI Modules and Task Scopes

## Purpose

This document lists the current AI task scopes and how they map to the shared overlay runtime.

## Current module model

AI features are not separate shell systems. They are task scopes registered against the shared overlay host.

- The overlay host is temporary and zero-footprint.
- The task registry decides which review renderer is used.
- The server decides what can be resolved, validated, and applied.

## Current scopes

### `mail-order-review`

Primary current mail scope.

- Used when the active context is an email thread.
- Optimized for "order from existing offer" flows.
- Produces a review with customer selection, reference document selection, warnings, and apply payload.
- Uses deterministic server resolvers for address and document candidates.

### `mail-classification`

Legacy/transitional scope.

- Kept for compatibility with older review components and older APIs.
- Should not be extended as the preferred path for new work.
- If retained, it must still go through server review and validation.

### `mail-to-document-draft`

Legacy/transitional scope.

- Represents draft proposal behavior for mail-driven document creation.
- Kept only while older UI or test code still references it.
- Prefer `mail-order-review` and dedicated resolver-driven flows instead.

### Generic discovery scopes

These are server-side discovery scopes, not UI business flows:

- catalog entities
- catalog commands
- catalog context
- context resolution

## Current module guidance

For new AI work:

- start from the active focus context,
- register a task scope,
- define the review schema,
- add deterministic resolvers,
- and keep apply logic behind a command.

Do not add new AI-only layout regions or one-off prompt branches when the shared runtime can handle the case.
