# Document Editor — Bug Tracker & Fix Plan

> Source: user analysis + live code review 2026-05-15.
> All 10 original user-reported issues are captured here first, then extended with code-level findings.

---

## From the User's Analysis (verbatim issues, numbered as reported)

### Issue 1 — New document form: N00 Belegart and Beleggruppe fields are wrong/broken

- [ ] When opening the new-document form for group N00, the Belegart dropdown shows all 13 types — user must re-select even though the group already determines the type. The group pre-selection from the sidebar click is not being honoured in the form correctly.
- [ ] Beleggruppe dropdown is only populated after Belegart is selected, but when arriving via F3 on a group node the type should already be locked and the group pre-selected — the two-step picker flow is wrong for the "arrive from sidebar" path
- [ ] The form should: lock Belegart to the group's `documentType`, pre-select the group, and go directly to address entry — not require the user to re-pick type and group they already navigated to

### Issue 2 — Entire interface is in German when user is in English

- [ ] `document-editor.tsx` has zero `useTranslation` calls — every string is hardcoded German
- [ ] Affected strings include all field labels (Rechnungsadresse, Lieferadresse, Lager, Zahlungsbedingung, Versandart, Datum, Währung, Belegart, Beleggruppe), table column headers (Pos, Artikel, Bezeichnung, Mge, Einh., EP, R%, MwSt%, Netto), all footer buttons (Schließen, Speichern, Buchen, Wandeln, Stornieren), and all inline messages
- [ ] `DocLookupField` dropdown also hardcoded: "Suchen...", "Keine Einträge", "Erst Belegart wählen"
- [ ] `ArticleSearchCell` placeholder "Artikelnr. / Bezeichnung..." hardcoded German
- [ ] Fix: `useTranslation('ui')` throughout `document-editor.tsx`; add `document.*` keys to `en.json` + `de.json`

### Issue 3 — Editor fills the entire screen

- [ ] `document-editor.tsx:1116`: `<div className="fixed inset-0 z-40 bg-canvas flex flex-col">` — fullscreen takeover
- [ ] Spec says: "Kein Modal-Dialog — dedizierte Detail-Pane im Workspace-Split. Voller vertikaler Raum für Header-Felder + Lines Editor"
- [ ] The intent is a right-pane split within the TriView workspace, not a fullscreen overlay that hides everything else
- [ ] Fix: integrate editor into the TriView right-pane slot (replaces InspectorPanel when a document is being edited); the list and sidebar remain visible on the left

### Issue 4 — Belegarten are completely wrong in the database

- [ ] All `seed.ts` inserts use `onConflictDoNothing()` — any existing rows with wrong names from a prior run are never corrected
- [ ] Canonical Belegart names that must be in DB: N=Angebot, A=Auftrag, L=Lieferschein, R=Rechnung, G=Gutschrift, b=Bestellung, l=WE-Lieferschein, r=WE-Rechnung, g=WE-Gutschrift, V=Inventurbuchung, Z=Zubuchung, E=Entnahme, U=Umlagerung
- [ ] Fix: replace `onConflictDoNothing()` with `onConflictDoUpdate({ set: { name: ... } })` on `documentType` rows so re-running seed corrects stale data
- [ ] Same applies to `documentGroup` name field — canonical group names must match

### Issue 5 — Lager (warehouse) is never pre-filled

- [ ] No default warehouse anywhere in the system — Lager field is always blank on new documents
- [ ] Required cascade per spec: `company.defaultWarehouseId` → `documentGroup.defaultWarehouseId` → `document.warehouseId` (overridable) → `documentLine.warehouseId` (overridable per line)
- [ ] `company` schema currently has no `defaultWarehouseId` column — this column is missing entirely
- [ ] `seedCanonicalDocumentGroups()` does not set `defaultWarehouseId` on any group
- [ ] `document-editor.tsx:885` tries `groupData.defaultWarehouseId ?? null` — result is always null because neither the group nor the company provides a default
- [ ] Fix (data): seed a default warehouse for the base tenant, add `defaultWarehouseId` to `company` table via migration, populate for base company
- [ ] Fix (logic): group-default fallback already coded; add company-default as secondary fallback before null

### Issue 6 — Direction (Richtung) is not pre-filled automatically

- [ ] `documentDirection` (OUTBOUND / INBOUND / ADJUSTMENT) is derived client-side via `deriveDirection()` in `document-editor.tsx:1345` and sent in the create payload — this part exists
- [ ] Problem: the field is invisible to the user — there is no Direction indicator in the editor header, so the user cannot verify or override it
- [ ] Bigger problem: when creating via the form's Belegart dropdown (not via group sidebar), `deriveDirection(header.documentType ?? "")` is called but `header.documentType` may be undefined at the point of save if the user never selected one, sending an empty string to the server
- [ ] The group fetch at `document-editor.tsx:819` provides `groupData.documentType` but this is only applied to `header.documentType` when group defaults are merged — race condition possible if save fires before group data loads
- [ ] Fix: derive direction server-side from `documentType` on the create endpoint — don't trust the client payload for this; remove `documentDirection` from the create body and compute it in the API handler

### Issue 7 — Keyboard navigation is not optimised for data entry

- [ ] **Header → Lines**: Tab from the last header field (Währung, tabIndex=7) falls through to browser default, does not enter the lines table. Required: Tab out of Währung → focus article field of first line (create line if none)
- [ ] **Lines → Next line**: Tab out of R% (Rabatt) field commits the current line and should jump to article field of next line (or create a new line if at the end). Code calls `addLine()` after 30ms — verify focus lands on `articleInputRef` of the new line via the `editingId` effect
- [ ] **Article search**: typing in article field opens dropdown; arrow up/down navigates; Enter selects and advances to Qty — this flow should work but needs verification that focus reliably lands on Qty after selection in all cases
- [ ] **Description field**: currently receives Tab focus between Article and Qty; it is auto-filled from article name and should be skipped by Tab (`tabIndex={-1}`)
- [ ] **End of document**: after the last line's R% field, Tab should create a new line, not escape the table
- [ ] **Lookup fields**: `DocLookupField` opens on Enter/Space/F5 — verify arrow+Enter selection advances Tab focus to the next header field correctly
- [ ] Fix: systematic Tab flow audit and implementation across header and lines; expose a `focusFirstLine()` imperative handle from `DocumentLinesEditor` via `useImperativeHandle`

### Issue 8 — Creating an N00 document for a customer shows nothing in addresses related documents

- [ ] `CustomerStatsSection` queries `/api/stats/address/:addressId` which reads `mv_sales_period_customer` — a materialized view built from `fact_sales_event`
- [ ] Type N (Angebot) produces no `fact_sales_event` entry (correct per posting matrix — N has no inventory or sales effect)
- [ ] Therefore N documents will never appear in the address stats section regardless of status
- [ ] The "last 10 documents" table must be a direct query: `SELECT * FROM document WHERE customer_id = :addressId ORDER BY created_at DESC LIMIT 10` — not derived from stats
- [ ] Fix: add `GET /api/addresses/:addressId/documents` returning recent documents for any type/status; wire `CustomerStatsSection` recent-docs table to this endpoint; keep stats endpoint only for revenue/profit figures

### Issue 9 — Overall performance is unacceptably slow

- [ ] No profiling data yet — but suspected causes:
  - `document-editor.tsx` fires 6+ parallel queries on mount (warehouses, paymentTerms, shippingMethods, currencies, taxCodes, document data, group data) — all fresh, no stale-time configured
  - `DocumentLinesEditor` issues a separate fetch per article pricing lookup (one round-trip per line on load)
  - The generic `/api/data/$` route in `data.ts` does no query result caching — every render cycle refetches
  - `useQuery` default `staleTime=0` means every focus/blur triggers a background refetch on all these queries
  - Materialized view queries on `mv_sales_period_customer` (stats) hit large aggregations even for empty result sets
- [ ] Fix: set `staleTime` on all reference-data queries (warehouses, currencies, payment terms, etc.) to at least 5 minutes — these change rarely
- [ ] Fix: batch the pricing lookups on document load into a single endpoint instead of N individual fetches
- [ ] Fix: investigate whether the generic data route is doing table scans due to missing indexes on `tenant_id + archived` combinations
- [ ] Action: run with React DevTools Profiler and network tab open; identify the top 3 slowest queries by time and fix those first

### Issue 10 — Wandeln duplicates the document instead of converting it

- [ ] `DocumentService.convertDocument()` (`packages/db/src/services/document-service.ts:605–706`) creates the target document and copies lines correctly
- [ ] Missing: the source document is **never archived** — no `UPDATE document SET status='archived'` is executed on the source after the target is created
- [ ] Result: original N document stays in `draft`, new A document appears — looks like a duplicate to the user
- [ ] Spec: *"setzt Quellbeleg auf `status = 'archived'`"*; *"Die Belegkette bleibt über `parent_document_id` und `transaction_id` vollständig nachvollziehbar"*
- [ ] Fix: inside the `db.transaction()` block in `convertDocument`, after inserting lines, add: `await tx.update(document).set({ status: 'archived', updatedAt: new Date() }).where(eq(document.documentId, documentId))`
- [ ] Also missing: `archivedAt` timestamp — add to the update if the column exists in schema

## Additional Bugs Found in Code Review

### Seed / Data Layer

- [ ] **Sidebar tree excludes all seeded groups** — `getDocumentTree()` at `document-service.ts:751` filters `.filter((g) => g.groupNumber > 0)` but seed creates all canonical groups with `groupNumber: 0`; sidebar is empty on a fresh install
- [ ] **Number sequences not seeded or linked** — `convertDocument` falls back to `DRAFT-{timestamp}` document numbers because `documentGroup.numberSequenceId` is null for all seeded groups; fix by seeding 13 `number_sequence` rows (prefixes: ANG-, AUF-, LIS-, RE-, GU-, BES-, WEL-, WER-, WEG-, INV-, ZUB-, ENT-, UMB-) and linking them to groups
- [ ] **Statistics materialized views never refresh** — `pg_notify('stats_refresh', tenantId)` is emitted at `document-service.ts:444` but no listener exists in the application; views are static; short-term fix: call `REFRESH MATERIALIZED VIEW CONCURRENTLY` inline after `postDocument` transaction

### Logic / Correctness

- [ ] **Storno creates negative-quantity lines** — `stornoDocument()` at line 517 uses `String(-Number(l.quantity))`; spec says *"identischen (positiven) Mengen"*; fix: `String(Math.abs(Number(l.quantity)))` — reversal comes from movement type G/g, not negative qty
- [ ] **`canConvert` true for posted documents** — `document-editor.tsx:1104`: `docStatus === 'draft' || docStatus === 'posted'`; convert is a draft-only workflow action; fix: `docStatus === 'draft'` only
- [ ] **Article column shows UUID fragment** — `document-editor.tsx:599`: `line.articleId?.slice(0, 8)` when not editing; should show `line.articleTextSnapshot` (article name already stored on the line)

## Architectural & Systemic Faults (Core Spec Violations)

- [ ] **Chatty API / Transaction Safety** — `saveMutation` iterates through lines in the frontend. If the loop breaks midway, the document is partially saved. Required: atomic `POST /api/documents/save` payload including header + lines (all-or-nothing transaction).
- [ ] **Missing Validation Issue Envelope Mapping** — Error handling uses generic toasts. Spec (§02) requires mapping the "Issue Envelope" back to specific fields and rows for visual feedback.
- [ ] **Missing Focus Manager Integration** — Editor tracks focus via local `editingId`. Should participate in platform Focus Manager to allow deterministic `Escape` handling and global command context awareness (§00).
- [ ] **Missing Alt+I (Context Statistics)** — No registration for the context statistics command. User cannot access KPI cards for the current customer/article without closing the editor (§01).
- [ ] **Address ID Decoupling** — UI explicitly sets `deliveryAddressId` to null when a snapshot is used. This destroys the link to master data, preventing historical reporting by address ID (§05).
